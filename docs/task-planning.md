# Task Planning

## 定位

Task Planning 的目标，不是写出一份好看的任务分解文档，而是把一个较大的目标拆成一组能够持续推进基线的 `Task`。

每个 `Task` 都是一个基线迭代增量。它以“当前最新基线 + 一个 Task Spec”为输入，无人值守地推进到一个独立 PR 被合并，再清理 WorkTree，最终让基线向前迭代一步。

因此，规划任务时最重要的不是目录怎么切、文件怎么分，而是：

- 如何把目标拆成多个有意义的基线增量。
- 如何最大化并行度。
- 如何避免让执行层在任务内部遇到人工决策节点。
- 如何让任务在基线变化时仍能通过 Assumptions 验证继续推进，或者尽早失败上报。

## 输入与输出

Task Planning 的输入通常包括：

- 一个较大的目标或问题陈述。
- brainstorming 访谈后的结论。
- 若干已经澄清的风险、冲突场景与价值排序。
- 当前仓库基线的关键现状。

Task Planning 的输出不是仓库内的 spec 文件，而是：

- 一组可批准的 `Task Spec`。
- 一张初始软依赖 DAG 快照。
- 一组可直接写入 SQLite `tasks` 表的结构化任务记录。

这里的 DAG 只是规划器时刻的初始视图，不承诺在运行时保持不变。后续执行层和基线验证流程可以根据真实情况重写各个 Task 的 `dependencies`。

在这个阶段，不要求先把 spec / implementation plan 提前写入 repo。只有当某个 Task 真正启动自己的 Task Session、创建了自己的 WorkTree 后，才在该 WorkTree 中产出 repo 内的正式 spec / implementation plan，并和实现一起进入同一个 PR。

## 核心原则

### 1. Task 是基线增量，不是讨论主题

Task 不应该写成“研究一下某件事”或“处理某个模块”。Task 必须天然对应一个可以被独立合并的 PR 主题，能够让基线向前推进一步。

### 2. 并行优先，但不是机械切碎

规划任务时，应优先寻找可以并行推进的增量窗口，而不是沿着实现顺序串行拆分。

但并行并不意味着越碎越好。过细的 Task 会带来：

- 管理开销上升。
- PR 数量膨胀。
- 测试粒度过细，变成实现细节的重复。
- 最后收口时出现大量隐性 wiring/integration 工作。

### 3. Task 尺寸围绕 10 到 30 分钟实现窗口设计

这里指的是 AI Agent 的主要实现与验证时长，而不是 Task 从创建到完全结束的 wall-clock 时间。PR checks、等待 merge、收尾清理等异步阶段可以更长。

一个合格的 Task 通常应满足：

- 主题单一。
- 可独立形成 PR 语义。
- 有 focused verification。
- 改动集中在一小簇强相关概念上。
- 可以被一个 Agent Session 独立闭环推进。

### 4. 依赖是软的，Assumptions 才是硬的

Task 之间的 `dependencies` 是软依赖提示集合，用来表达：当前哪些前置任务最可能让当前 Task 的 Assumptions 成立。

规划器可以给出一个初始值，但后续允许执行层或基线验证流程完全重写它。

Task 在运行时是否真正 ready，不以依赖是否全部“完成”为准，而以最新基线是否能通过该 Task Spec 的 `Assumptions` 验证为准。

因此：

- 一个 Task 可以在生命周期内多次被基线验证卡住。
- 它可以自己识别当前未满足的 Assumptions。
- 它可以读取环境中的其他 Task，推断哪些 Task 最可能推动这些 Assumptions 成立。
- 它可以把当前的候选前置任务重新写回 `dependencies`，作为后续调度提示。

Task Planning 要接受这个现实：规划出来的 DAG 不是硬阻塞图，而是一个对“可能的推进顺序”的高质量初始猜测。

## 如何拆 Task

### 1. 先找目标状态，不要先看文件

先回答：基线最终要多出哪些概念、能力、契约或关系变化？

不要一上来就想“改哪个目录”或“拆几个文件”，而要先看哪些基线增量可以独立成立。

### 2. 先找最小地基，再找并发窗口

拆分时，优先找那些会阻塞后续并发的最小共享地基，例如：

- 新的 contract 或 schema
- 新的 shared type
- 新的 route skeleton
- 新的 mock shape
- 新的任务状态 / 调度契约

这些地基 Task 应尽量小，只提供后续并发所必需的前提，不要把大量后续工作提前塞进去。

地基之后，再找能够并行推进的多个主题切片。

### 3. 让每个 Task 只承载一个主题

如果一个 Task 同时承担“定义契约 + 落后端能力 + 做前端消费 + 补文档”，它通常就太粗了。

一个 Task 只应承载一个清晰主题，例如：

- 扩展某个 contract 形状
- 落地某个 endpoint
- 实现某个 selector / adapter
- 实现某个 widget / section shell
- 完成一次 wiring

### 4. 尽量把高依赖节点推迟到 DAG 后段

高依赖节点不是不能存在，但应该尽量收敛到后段，并且职责必须薄。

典型地可以把节点心智分成：

- `Base`: 提供共享地基，应接近零依赖。
- `Producer`: 产出真实能力，应尽量低依赖以获得并行度。
- `Consumer`: 消费契约或 mock shape，也应尽量低依赖。
- `Wiring`: 把已有能力接起来，允许高一些依赖，但只能做接线。
- `Integration`: 做回归、文档、少量验收修正，必须很薄。

如果最后出现一个巨大的 wiring 或 integration 节点，通常说明前面的拆分失败了。

## 如何建立任务依赖

任务依赖表示：当前 Task 认为，哪些前置任务的基线增量最可能帮助自己的 Assumptions 成立，或者显著降低返工概率。

依赖不是任务相关性的表达，也不是硬门禁，而是对基线前置条件的软提示表达。

### 允许建立依赖的场景

- 语义前置依赖：当前任务明确依赖某个新概念、新契约、新接口面已经进入基线。
- 共享地基依赖：某个最小地基必须先建立，后续多个任务才能稳定并行。
- 冲突规避依赖：两个任务高概率会频繁争用同一个共享面，串行化更便宜。

### 不该建立依赖的场景

- 仅仅因为属于同一功能。
- 仅仅因为最终要集成。
- 仅仅因为“先做完 A 更容易理解 B”。
- 仅仅因为都在同一个目录下。
- 仅仅因为 planner 想让图更整齐。

### 软依赖的规划含义

软依赖的作用不是阻止 Task 启动，而是：

- 帮助上层 AI 组织一个更合理的初始 DAG。
- 帮助调度器优先重试最可能变 ready 的 Task。
- 帮助执行层在 Assumptions 未满足时持续维护“我现在更该盯着谁”。

因此，依赖应尽量稀疏，只保留真正有信息量的边。规划器的职责是给出高质量初始值，而不是垄断它的后续解释权。

## Task Session 与运行时现实

Task Planning 必须面向真实运行时，而不是理想化串行流程。

在运行时：

- 所有 Task 都可以被视为同时存在于调度图里。
- 某个 Task 可能多次进入“验证未通过 -> 等待 -> 再验证”的循环。
- 当前真实阻塞原因不是“某依赖没 done”，而是“当前基线还未满足该 Task Spec 的 Assumptions”。
- Task Session 允许自己查看环境中的其他 Task，推断哪些 Task 可能推动当前 Assumptions 成立。

因此，Task Planning 不需要试图把所有运行时顺序写死。更重要的是：

- 给每个 Task 一个高质量、可执行的 Spec。
- 把明显有帮助的软依赖写出来。
- 避免规划出互相强冲突的大量并行 Task。

## 规划流程

推荐按以下顺序规划：

1. 明确要推进到的目标状态。
2. 提炼当前基线中的核心概念、缺口和约束。
3. 找出最小共享地基。
4. 找出可并行推进的主题切片。
5. 为每个切片撰写独立的 `Task Spec`。
6. 只在真正必要时建立软依赖。
7. 检查是否产生巨型 Base / Wiring / Integration 节点。
8. 检查每个 Task 是否仍处于 10 到 30 分钟的实现窗口内。
9. 审视图中是否存在明显会互相争用同一共享面的并行 Task。
10. 批准后，将 Task 批量写入 SQLite。

## 规划评审清单

一个 Task Planning 结果应至少通过以下检查：

- 每个 Task 都能自然映射成一个独立 PR 主题。
- 每个 Task 都有清晰的 focused verification。
- 不存在为了“好理解”而额外添加的大量串行依赖。
- Consumer 只要能基于 contract 或 mock shape 开发，就不要强依赖真实 backend。
- Wiring 节点只做接线，不继续扩 scope。
- Integration 节点足够薄，不承接大块遗漏开发。
- 依赖图整体能最大化并行，而不是把风险都提前塞进单一地基节点。
- 每个 Task 都能在无人值守前提下独立失败、独立成功、独立上报。

## 输出形式

规划器的输出最好同时具备两种视角：

- `Task Graph`: 展示节点、软依赖、并行窗口。
- `Task Cards`: 展示每个 Task 的标题、类型、为什么它是独立主题切片、前置依赖、验证重点与 out-of-scope。

但这些结构化结果的持久化真相，应优先进入 SQLite，而不是主要依赖 Markdown 文档。

Markdown 更适合辅助理解；SQLite 更适合调度、查询、恢复和推进。
