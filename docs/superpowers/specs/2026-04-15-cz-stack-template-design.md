# CZ-Stack 模板基础设施设计说明

## 背景与目标

CZ-Stack 需要一份面向模板基线的设计说明，用于约束后续实现范围，并明确模板要解决的共性问题。该模板的定位是 **Balanced core + optional modules**：先提供一套足够强、可直接落地的核心基础设施，再通过按需模块扩展覆盖不同项目阶段，而不是把所有能力一次性塞进模板。

本设计的目标覆盖三类核心交付形态：**Web 前端应用、Hono API 服务、oclif CLI 工具**。三者围绕同一套 TypeScript 基线、协议契约、测试策略、CI 约定与发布约定协同工作，形成适合新项目启动与持续演进的统一模板。

## 设计原则

### 1. 强约定基础设施，弱约定业务布局

模板对基础设施做强约定，以减少工具链分叉、降低 agent 与团队协作成本；对业务模块布局保持弱约定，允许项目按实际边界组织目录与模块。

强约定范围如下：

- **pnpm workspace** 作为仓库级依赖与脚本编排基础。
- **TypeScript only**，不为多语言或混合语言场景提供一等路径。
- Web 端采用 **Vite + React**。
- API 服务采用 **Hono**。
- 协议层采用 **OpenAPI-first + Zod**。
- 统一使用 **Biome** 完成格式化与静态检查。
- 通用库与可发布包采用 **tsdown** 构建。
- 版本与发布采用 **Changesets**。
- 测试采用 **Vitest + Playwright + CLI smoke** 三层体系。
- CI 采用 **release-aware** 设计，覆盖从验证到发包前检查的完整链路。

### 2. 核心能力完整，增强模块按需启用

模板默认提供可启动项目所需的核心能力，但数据库访问实现、SDK 生成、增强模块组合等能力保持可选，不强制所有项目预装。这样既避免模板臃肿，也能保留面向不同产品类型的适配空间。

### 3. 契约先行，模块围绕同一事实源协作

协议契约是前后端、CLI、文档站之间的共享边界。模板应通过单一契约中心减少接口漂移，使 API 文档、类型约束、测试输入输出和未来 SDK 生成都围绕同一份定义工作。

### 4. 可替换实现优先于预绑定实现

对于数据库、模块目录、是否生成 SDK 等高差异化能力，模板只定义稳定边界与默认路线，不预先绑定单一实现，更不将 ORM、固定目录结构或所有增强能力写死进模板。

## 模块角色与仓库组织

模板不强制使用 `apps/`、`packages/`、`services/` 这类固定目录名，而是鼓励按模块角色组织仓库，使结构能直接表达职责。典型角色包括但不限于：

- `web-app`：面向浏览器的 React + Vite 应用。
- `api-service`：基于 Hono 的 API 服务。
- `cli-tool`：基于 oclif 的命令行工具。
- `contract-package`：协议契约中心。
- `shared-lib`：可复用通用库。
- `docs-site`：文档与 API 展示入口。
- `db-adapter`：数据库访问边界与适配实现。
- `tooling-package`：共享脚本、配置或工程工具封装。

该组织方式的约束重点不在目录名字，而在模块角色清晰、依赖方向明确：业务模块依赖共享契约与通用能力，公共配置尽量集中，具体模块根据需要做局部覆盖。

## 协议层设计

`contract` 模块是模板中的协议中心，采用 **OpenAPI + Zod** 双表达协同模式：

- OpenAPI 作为跨模块共享的接口描述事实源。
- Zod 作为运行时校验与 TypeScript 类型推导的约束载体。
- Web、API、CLI 与文档站都围绕这份同源契约消费能力。

协议层的设计要求如下：

1. API 服务的路由输入输出应与契约保持同源约束，避免服务实现与文档脱节。
2. Web 应用优先基于契约消费接口类型与错误模型，而不是手写并重复维护类型。
3. CLI 工具调用 API 或消费共享领域模型时，应复用同一契约定义，避免出现独立协议分支。
4. API 文档必须由契约同源生成与展示，文档站只是展示入口，不是第二份手写规范。
5. 是否生成 SDK 属于可配置选项；模板应允许围绕契约生成 SDK，但不把“必须生成 SDK”作为基线要求。

## 数据与持久化边界

数据库策略采用 **SQLite-first, flexible adapter**。其含义是：

- 模板默认优先考虑 SQLite 作为启动项目、原型验证与轻量部署的首选路径。
- 模板只预留 `db` 访问边界与适配层，不绑定 ORM，也不把具体数据库访问实现写死。
- 项目可在该边界后替换为不同的 adapter、查询层或 ORM，但这属于后续项目选择，不属于模板基线强约定。

因此，模板在数据层只定义“如何接入”和“边界放在哪里”，而不定义“必须使用哪种 ORM / migration 工具 / 查询风格”。这样可以兼顾初始落地效率与后续演进弹性。

## 工具链、测试与发布

### 工具链

模板基础工具链固定为：**pnpm workspace + TypeScript + Biome + tsdown**。

其中：

- pnpm workspace 负责多模块依赖管理与统一脚本入口。
- TypeScript 作为唯一一等语言与类型系统基础。
- Biome 负责格式化、lint 与基础一致性治理。
- tsdown 用于共享库、契约包、工具包等通用模块构建。
- Vite 保持自身的应用构建链，Web 应用不需要为了统一而脱离 Vite 原生能力。

### 测试

模板测试采用三层结构：

- **Vitest**：覆盖单元测试、轻量集成测试与共享库校验。
- **Playwright**：覆盖 Web 端关键交互与跨模块主流程验证。
- **CLI smoke**：覆盖命令行工具最小可运行性、关键参数与主路径执行结果。

该设计强调不同交付形态都有对应的最小验证层，同时避免将所有测试职责堆到单一框架中。

### 发布与 CI

版本与发布采用 **Changesets**，原因是其更适合多包仓库、agent 辅助生成 release notes，以及按模块执行发包与版本编排。

CI 应具备 release-aware 能力，至少包含以下阶段：

- install
- typecheck
- lint
- test
- build
- OpenAPI 校验
- changeset 校验
- CLI smoke
- Playwright
- 发包前检查

CI 的目标不是只验证“能否通过构建”，而是同时保障契约、发布元数据与各交付形态的最小质量门槛。

## 文档与配置策略

### 文档

模板应至少提供以下文档入口：

- README：面向仓库使用者说明模板定位、快速开始与模块概览。
- 架构与模块角色说明：解释模块边界、依赖方向与可选增强能力。
- API 文档展示入口：面向契约消费者展示同源生成的 API 文档。

文档目标是帮助新项目快速理解模板，而不是把所有内部实现细节提前写死。

### 配置策略

配置策略遵循“**公共配置集中，具体模块按需覆盖**”：

- 仓库级配置集中管理，如 TypeScript、Biome、测试与发布基础约定。
- 模块级配置只在确有差异时覆盖，不鼓励每个模块复制一套完整配置。
- 模板对基础设施强约定，但对业务目录布局、模块组合、数据库实现、SDK 生成与增强模块启用保持可配置。

## 不追求与非目标

本模板明确 **不追求** 以下方向：

- 不做 Bun-first 路线。
- 不服务多语言或混合语言模板诉求。
- 不强绑定任意 ORM。
- 不强制固定目录结构。
- 不一次性内置所有能力。

进一步的非目标为：

1. 不服务所有未来技术选型，模板只覆盖当前明确认可的技术基线。
2. 不保证同时适合所有类型产品，模板是高复用起点，而不是无限泛化的平台。
3. 不在本设计中展开具体实现、脚手架细节、模块内部 API 或数据库技术选型细节。

## 验收标准

当后续实现严格满足以下条件时，可视为符合本设计：

1. 仓库采用 pnpm workspace，多模块共享 TypeScript only 基线与公共工程配置。
2. Web、API、CLI 三类模块均有明确角色定位，且可围绕共享契约协同工作。
3. 协议层存在单一 `contract` 中心，能够支撑 OpenAPI、Zod、文档展示与多消费者复用。
4. 数据层仅定义 SQLite-first 的默认方向与 db adapter 边界，不预绑定 ORM。
5. 工具链体现 Biome、tsdown、Changesets、Vitest、Playwright、CLI smoke 与 release-aware CI 的统一约定。
6. 仓库组织允许按模块角色命名，不要求固定使用 `apps/packages/services` 目录名。
7. 文档层至少覆盖 README、架构/模块角色说明与 API 文档展示入口。
8. 可配置项与非目标边界清晰，未把目录结构、数据库实现、SDK 生成或增强模块错误提升为强制要求。
