# Node.js 24 LTS 仓库基线升级设计说明

## 背景/目标

当前仓库的 Node.js 基线仍以 22 为主：根 `package.json` 的 `engines.node` 仍声明 `>=22.0.0`，复用的 `.github/actions/setup-node-pnpm/action.yml` 仍固定 `node-version: 22`，后续实现还需要继续确认 README、开发文档与本地版本声明文件是否存在与是否同步。与此同时，仓库中的 `@types/node` 已升级到 24 系列，说明工具链已部分向 Node 24 靠拢，但仓库基线口径尚未统一。

本次任务的目标是把 monorepo 的默认 Node.js 运行基线统一提升到 Node.js 24 LTS，并把这件事限制为一次“基线升级 + 最小兼容性修正”任务，而不是借机进行无关的工具链现代化改造。后续实现应先统一所有显式 Node 基线声明，再在 Node 24 环境下执行完整验证，仅在验证暴露真实失败时补充最小必要兼容修复，最后完成 README / 开发文档口径同步。

## 范围

本次设计约束的范围如下：

1. 将仓库中由项目自身维护的 Node.js 基线声明从 22 统一到 24 LTS。
2. 更新 CI 中与 Node.js 版本相关的基线配置，使默认验证环境切换到 Node 24。
3. 如仓库已存在 `.nvmrc`、`.node-version` 一类本地开发版本声明文件，则同步到 24；若不存在，仅在后续实现确有必要且符合仓库惯例时最小化补充。
4. 在 Node 24 下运行仓库既有验证命令，并仅对 Node 24 直接引发的兼容性问题做最小修复。
5. 更新 README 与开发者文档，确保不再把 Node 22 表述为当前默认基线。

## 非目标

本次任务明确不包含以下内容：

1. 不扩展到 Bun、Deno 或多运行时支持策略。
2. 不借机升级与 Node 24 无直接关系的依赖、构建工具或工程体系。
3. 不进行与 Node 24 兼容性无关的重构、目录调整或脚本重写。
4. 不为了“顺便清理”而修改未受 Node 24 影响的测试、文档或 CI 结构。
5. 不把本次任务扩大为全面平台现代化项目。

## 方案设计

### 1. 先统一仓库显式 Node 基线声明

后续实现必须优先收敛仓库中由项目直接维护的 Node.js 版本声明，形成单一事实来源，避免“依赖已经支持 Node 24，但仓库入口仍宣称 22”为主的混合状态。这里关注的是仓库自有配置，而不是 lockfile 中第三方依赖声明的 `engines` 字段。

预期重点检查并更新以下位置：

- 根 `package.json` 的 `engines.node`。
- `.github/actions/setup-node-pnpm/action.yml` 中的 `actions/setup-node` 版本配置。
- 仓库内已存在的 `.nvmrc`、`.node-version` 等本地开发版本文件（若存在）。
- README、开发说明、验证文档中显式写死 Node 22 的内容（若存在）。

统一原则是：仓库默认基线应明确指向 Node.js 24 LTS，不再保留“22 为默认、24 可选”这一过渡口径。

### 2. CI 基线切换只做版本对齐，不额外重构 workflow

CI 升级的核心是让仓库主要验证路径在 Node 24 上运行，而不是借此重写 workflow 结构。后续实现应以当前复用的 `.github/actions/setup-node-pnpm` 为主要切入点，把其 `node-version` 调整到 24，使引用该 action 的 workflow 自然继承新的 Node 基线。

除非在实现时发现某个 workflow 存在额外写死的 Node 版本，否则不应扩展到无关 job 拆分、缓存策略重构或 CI 编排调整。也就是说，Node 基线升级是本次 CI 变更的唯一目的；如果 CI 在 Node 24 下失败，优先修复兼容性根因，而不是重构验证流水线。

### 3. 兼容性修复必须以“Node 24 暴露真实失败”为触发条件

本次任务允许进行代码、脚本或配置修正，但前提是 Node 24 环境下的既有验证实际暴露了问题。后续实现必须按以下顺序推进：

1. 先统一版本声明与 CI 基线。
2. 再在 Node 24 下执行完整验证。
3. 仅当验证失败且根因可归因于 Node 24 运行时、Node 24 相关依赖约束或 Node 24 下工具行为差异时，才做最小修复。

允许的修复类型包括：

- 因 Node 24 API / 行为变化导致的脚本兼容性修复。
- 因 Node 24 下工具链（如 Playwright、oclif、tsx、tsdown 等）真实不兼容而必须进行的最小配置调整或依赖补丁。
- 因 Node 24 引发的类型、运行时或测试入口错误的最小修正。

不允许的做法包括：

- 在未复现失败前，预防性升级一批无关依赖。
- 以“兼容性优化”为名扩大到 unrelated refactor。
- 把本可局部修复的问题上升为体系重写。

### 4. 文档同步要围绕“默认开发基线”统一口径

README 与开发者文档的职责是准确反映仓库当前默认环境，因此后续实现必须排查并修正文档中与 Node 基线相关的叙述，保证新的读者不会继续按 Node 22 准备环境。

文档更新重点包括：

- 安装前提、开发环境要求、快速开始说明中的 Node 版本描述。
- 任何把 Node 22 作为当前推荐版本、最低版本或 CI 基线的表述。
- 与本地版本管理工具相关的说明（如 nvm / nodenv / asdf 等），若仓库已有对应说明则需同步。

文档同步应只服务于 Node 24 基线升级本身，不应顺带重写 README 结构或补充无关教程。

### 5. 验证口径保持仓库既有主路径

后续实现完成后，应在 Node 24 环境下优先执行仓库现有主验证入口，确认升级不是停留在声明层面，而是实际可运行。验证目标如下：

1. `pnpm lint`
2. `pnpm typecheck`
3. `pnpm test`
4. `pnpm build`
5. `pnpm smoke`

若某条命令因 Node 24 升级暴露失败，应先定位最可能根因并做最小修复；若失败原因经确认超出本 spec scope，则应显式记录阻塞，而不是擅自扩大范围。

## 影响范围

预期受影响文件集合应以最小必要范围为准，通常包括：

- `package.json`：统一根级 `engines.node`。
- `.github/actions/setup-node-pnpm/action.yml`：统一 CI setup action 的 Node 版本。
- `.github/workflows/ci.yml` 或其他 workflow：仅在存在额外写死版本时同步收敛。
- `.nvmrc`、`.node-version`：若仓库已有则同步；若无，则按“确有必要且符合惯例”原则决定是否新增。
- `README.md` 与相关开发者文档：同步默认基线说明。

lockfile 中第三方包的 `engines` 元数据不属于“仓库自有 Node 基线声明”，不应为了数字统一而手工编辑；只有在后续实现因依赖变更而自然更新 lockfile 时，才接受其随变更调整。

## 风险与控制

本次升级的主要风险如下：

1. 某些工具或依赖在 Node 24 下出现尚未暴露的兼容性问题，例如 Playwright、oclif、tsx、tsdown 等。
2. 仓库内存在遗漏的 Node 22 文档或版本声明，导致口径仍然不一致。
3. CI 与本地开发环境口径不一致，造成“本地可过、CI 不过”或反之的偏差。

对应控制策略：

- 先统一显式声明，再跑完整验证，避免一边修问题一边猜基线。
- 以搜索和 diff 方式检查 Node 22/24 文案，减少文档漏改。
- 兼容性修复只围绕真实失败点展开，避免过度升级带来新的变量。

如升级后出现无法在当前 scope 内解决的问题，可临时回退相关兼容补丁或基线声明，但该回退应以阻塞记录为前提，而不是在实现阶段预设保守不升级。

## 验收标准

当后续实现满足以下条件时，可视为符合本设计：

1. 仓库中所有由项目维护的默认 Node 基线声明已统一指向 Node.js 24 LTS。
2. 主要 CI 路径已在 Node 24 上运行，不再以 Node 22 作为默认 setup 基线。
3. README 与相关开发文档已不再把 Node 22 表述为当前默认环境。
4. 在 Node 24 环境下，`pnpm lint`、`pnpm typecheck`、`pnpm test`、`pnpm build`、`pnpm smoke` 作为主验证口径达到通过，或对超 scope 阻塞有明确记录。
5. 除 Node 24 直接引发的问题外，本次变更未扩展到无关的大规模重构或工具链现代化。

## 实施边界提醒

后续实现必须始终以“统一 Node 24 LTS 基线并修复直接受其影响的问题”为边界。任何与该目标无直接关系的依赖升级、脚本重写、CI 架构调整或文档重构，都应视为 scope drift。若实现过程中发现必须越过该边界才能继续推进，应先升级决策，而不是自行扩大改动面。
