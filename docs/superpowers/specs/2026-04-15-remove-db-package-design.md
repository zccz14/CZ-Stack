# 移除 `@cz-stack/db` 模板基线包设计说明

## 背景

当前 CZ-Stack 首版模板把 `modules/db` / `@cz-stack/db` 作为基线模块之一，用于承载 SQLite-first 的数据库边界与默认适配实现。这个设计虽然强调了“默认使用 SQLite、不要过早绑定 ORM”，但也在模板层引入了一个额外的共享数据层抽象，使数据库接入路径看起来像是模板必须先有一个独立 db 包，再由后端服务去消费它。

该基线不再符合当前模板定位。CZ-Stack 需要保留低运维、低依赖、快速启动的默认姿态，但不再鼓励“一套共享大数据库基线 + 一个专门 db adapter 层”成为模板默认结构。数据库接入应回到后端服务自身，例如 `modules/api`，并按实际业务边界直接组织。

## 目标

本设计用于约束一次模板基线收敛，核心目标如下：

1. 从模板基线中删除独立的 `modules/db` 包与 `@cz-stack/db` 包定位。
2. 明确数据库访问属于各后端服务自己的职责，而不是模板统一提供的共享内部抽象层职责。
3. 保留 SQLite-first 作为默认原则，但把它表达为“后端服务默认优先采用嵌入式 SQLite 的落地姿态”，而不是“模板默认自带一个 SQLite-first db 包”。
4. 在 README、架构文档、验证文档与后续实现说明中，同步移除 dedicated db package 的测试、脚本、模块角色与能力表述。
5. 明确保留更重数据库方案的可选性，例如 Postgres，但其适用前提应是规模、部署约束或业务要求，而不是模板默认姿态。

## 设计原则

### 1. SQLite-first 仍是默认，但默认落点改为服务内

模板仍然优先面向低运维、低依赖、快速启动的新项目场景。因此，后端服务默认应优先考虑 SQLite 这类嵌入式存储，以减少额外基础设施依赖、提升本地启动速度，并让模板在单机、原型验证、小团队协作与轻量部署环境中更容易落地。

这里的 SQLite-first 是模板对默认落地姿态的选择，不是对统一共享 db 包的要求。

### 2. 数据库连接应按服务边界直接组织

模板明确采用以下架构立场：数据库通常应直接在各后端服务内部接入，并由服务根据业务边界维护自身的数据访问代码、连接配置、schema 组织与运行时装配。

这意味着模板不再鼓励以下默认结构：

- 先建立一个跨服务共享的“大数据库基线”。
- 再建立一个专门的内部 db adapter 包，要求各服务统一穿过该层访问数据库。

若未来具体项目确有多个服务共享同一种数据库访问基础设施的明确需求，可以在项目内按实际需要演进出共享库；但这不是 CZ-Stack 模板基线在本次任务中要提供或预留的默认抽象。

### 3. 不引入新的内部数据库抽象层替代 `@cz-stack/db`

移除 `modules/db` 的同时，不得在模板基线中新增另一个承担相同职责的内部包、共享 helper 层或“轻量版 db boundary”作为替代。否则只是换名保留原有结构，无法真正完成基线收敛。

本次设计的方向是减少模板内建抽象，而不是把 `@cz-stack/db` 迁移为另一个名字或另一种包装形式。

### 4. 重型数据库仍被允许，但不再构成默认心智

当项目规模、并发要求、运维边界、托管约束或跨实例一致性需求要求使用 Postgres 等更重数据库时，模板允许服务直接接入这些方案。模板文档应明确这一点，避免把 SQLite-first 误读为“只能使用 SQLite”。

但模板默认叙述必须反映清楚：更重数据库是按需升级路径，不是首版模板的默认姿态，也不是必须预装在基线中的基础设施。

## 设计变更

### 1. 基线模块集合调整

模板基线模块集合从“contract + api + web + cli + db”调整为“contract + api + web + cli”。

这项调整意味着：

- `modules/db` 不再作为首版核心模块存在。
- `@cz-stack/db` 不再作为 README、架构文档、模块角色文档中的基础能力入口出现。
- 数据访问相关说明应下沉到后端服务职责描述中，而不是保留一个独立模块角色。

### 2. 后端服务职责说明调整

`api-service` 一类后端服务的职责应扩展为：基于 contract 暴露 HTTP API、组装运行时依赖，并在服务内部直接管理数据库接入与持久化实现。

文档应把数据库访问表述为后端服务的内聚职责之一，而不是额外依赖一个模板预置 db 层。后端服务内部可以按实际复杂度决定是否拆分 repository、query、schema、migration、driver setup 等文件，但这些组织方式属于服务实现细节，不再上升为模板级共享包约定。

### 3. 文档叙述调整

模板文档需要统一转向以下表述：

1. 默认原则：后端服务优先采用 SQLite-first 的嵌入式存储路线，以降低模板启动与部署门槛。
2. 架构立场：模板不鼓励“共享大数据库基线 + 专门 db adapter 层”的默认结构。
3. 可选升级：当业务需要时，服务可直接接入 Postgres 等数据库，不受模板基线限制。

因此，后续实现中应删除或改写所有会传达以下错误心智的文档内容：

- “模板首版包含独立 db 模块。”
- “SQLite-first 的默认落点是一个共享 db 包。”
- “所有服务都应先经过一个模板内置 db adapter 再访问数据库。”

### 4. 测试、脚本与验证入口调整方向

本 spec 不直接实施代码修改，但要求后续实现遵循以下方向：

- 移除专门针对 `modules/db` 的包级测试、构建脚本与验证入口。
- 更新仓库级测试说明，避免继续把 `db` 视为独立验证对象。
- 若数据库相关验证仍有必要，应由具体后端服务在其自身测试中承担，而不是保留 dedicated db package 测试层。
- README、架构文档、验证文档、Changesets 说明与其他模板描述中，凡是把 `@cz-stack/db` 作为首版基线能力、独立模块或固定验证入口的内容，都应在后续实现中同步清理。

## 影响范围

本设计约束的影响范围包括但不限于：

- `README.md` 中的模板定位、模块入口、能力清单与验证说明。
- `docs/architecture/module-roles.md` 中的模块角色定义、允许依赖方向与禁止反向依赖规则。
- `docs/architecture/repo-conventions.md`、`docs/architecture/validation.md` 等架构说明文档。
- 任何 implementation plan、脚本、测试、Changesets 元数据或包清单中对 `modules/db` / `@cz-stack/db` 的基线引用。

本设计不要求在 spec 阶段枚举所有受影响文件，但后续实现必须以“移除模板基线中的独立 db 包心智”为判断标准，确保清理动作覆盖文档、脚本与验证口径，而不只是在目录层面删除一个包。

## 非目标

本设计明确不包含以下内容：

1. 不为模板新增新的数据库共享抽象层、数据库基础库包或内部 adapter 框架。
2. 不在本次任务中规定 API 服务必须采用哪种 ORM、query builder、migration 工具或 schema 管理方案。
3. 不禁止未来具体项目按需使用 Postgres、MySQL 或其他数据库。
4. 不把数据库能力推广到前端、CLI 或 contract 层，使其感知具体数据库实现。
5. 不在本次任务中扩展出多服务数据治理策略、跨服务共享 schema 机制或统一 repository 模式。

## 验收标准

当后续实现满足以下条件时，可视为符合本设计：

1. 仓库基线中已不再存在 `modules/db` 包，也不再把 `@cz-stack/db` 视为模板核心模块。
2. 模板文档已经明确 SQLite-first 仍是默认原则，但默认落点是后端服务内部，而不是共享 db 包。
3. 模板文档已经明确不鼓励“共享大数据库基线 + 专门 db adapter 层”的默认架构。
4. 模板文档已经明确 Postgres 等更重数据库仍被允许，但属于按需选择而非默认姿态。
5. dedicated db package 的测试、脚本、验证入口与文档引用已在后续实现中移除或改写为服务内责任。
6. 模板未引入任何新的内部 db 抽象层来替代 `@cz-stack/db`。

## 实施边界提醒

后续实现只能围绕“移除模板基线中的独立 db 包，并把数据库职责收回后端服务”展开，禁止借机扩展到新的数据层框架设计、ORM 选型争论或额外基础设施重构。若实现阶段需要新增数据访问约定，必须先确认是否超出本 spec scope。
