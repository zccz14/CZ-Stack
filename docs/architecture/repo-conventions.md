# 仓库组织约定

CZ-Stack 采用“**模块角色优先**”的仓库组织方式：我们关心的是模块承担什么职责、遵守什么依赖方向，而不是强制使用固定目录名。

当前首版基线已经在 `modules/` 下落地 `contract`、`api`、`web`、`cli` 四个模块，因此本约定描述的是**当前仓库现实**与后续演进规则，而不是与实现脱节的抽象建议。

## 角色优先，不要求固定目录名

- 不要求必须使用 `apps/`、`packages/`、`services/` 等传统分层目录名。
- 只要 workspace 发现规则、依赖方向与角色边界成立，模块可以放在任意等价物理目录中。
- 文档、脚本与配置应描述“角色约束”，而不是把目录名错误提升为架构本身。
- `contract` 必须保持协议中心地位；无论未来移动到何处，都不能破坏 API / Web / CLI 对这一事实源的依赖方向。

这意味着：

- `web-app` 不因为位于 `modules/`、`apps/` 或其他目录而改变角色。
- `contract-package` 的关键不是物理路径，而是它是否仍然作为协议单一事实源。
- `api-service`、`cli-tool` 等模块都按角色职责判断，而不是按目录前缀判断。

当前依赖方向应继续保持为：

```text
contract <- api
contract <- web
contract <- cli
```

其中：

- `contract` 是中心，但不是运行时组装层。
- 数据库接入默认由 `api` 等后端服务在各自服务内部组织，不预置共享数据库边界层。
- 文档说明只描述如何消费由 contract 同源生成的结果，不新增第二份协议事实源，也不在仓库内内置 docs 项目。

## 首版为何选择 `modules/`

首版示例选择 `modules/` 作为主要物理容器，原因如下：

1. `modules/` 比 `apps/` / `packages/` / `services/` 更中性，能同时容纳应用、服务、契约与工具模块。
2. 它能提醒维护者：这里承载的是“不同角色模块”，而不是某一种固定分层范式。
3. 在模板早期阶段，使用一个统一容器可以减少 workspace 配置分散与目录命名争议。

但这只是**首版示例容器**，不是永久强约束。若未来迁移到其他目录结构，只要同步更新 workspace globs、文档链接与脚本入口，并继续满足角色边界，就不构成架构偏离。

同理，`modules/` 的存在也不意味着未来必须把所有增强能力都放进同一目录；关键仍是角色边界与依赖方向，而不是路径字面量。

## 与 workspace 规则的关系

当前仓库使用 `pnpm-workspace.yaml` 声明首版发现规则，例如：

- `modules/*`
- `tooling/*`

这些规则的作用是让首版模板可直接安装、检查与扩展，而不是宣告目录名不可变。后续若目录演进，应优先确保：

1. workspace 仍能发现对应模块；
2. 文档入口与命令示例保持一致；
3. 模块角色与禁止反向依赖规则不被破坏。

如果目录演进会影响 `pnpm build`、`pnpm test`、`pnpm openapi:check`、`pnpm smoke` 等验证脚本，则脚本与文档必须同步更新，保证验证入口仍反映仓库现实。

同时，workspace 发现规则不代表这些路径下已经存在已实现模块；它们只是为后续任务预留的发现入口。

当前仓库中 `modules/*` 下的核心模块已经存在并参与验证，因此本文档不得再把它们描述成“仅预留”。

## OpenAPI 文档源约定

- `modules/contract/openapi/openapi.yaml` 是仓库内唯一 OpenAPI 事实源。
- 如果部署环境对外提供 `/openapi.yaml`，它只是这份 YAML 的发布地址。
- `/openapi.json` 仍可作为运行时导出结果存在，但它是消费格式，不是事实源。
- README、`docs/api/README.md` 与其他架构文档都只能围绕这条口径说明现状。

## 非目标与边界提醒

- 非目标：把 `modules/`、`tooling/` 等路径约定升级为不可变架构规则。
- 非目标：在文档里承诺超出当前实现的 ORM、SDK 生成器、内置文档站点或额外交付形态。
- 边界：SQLite-first 只说明后端服务默认优先采用嵌入式存储，不排斥 Postgres 等按需升级。
- 边界：可选 SDK 生成必须围绕 contract 扩展，不能反向驱动 contract 为生成器私有需求失真。

## 维护约定

- 新增模块时，先声明其角色，再决定其物理位置。
- 调整目录时，先更新 workspace 配置与文档，再移动实现。
- 若某目录命名会误导角色理解，应优先修正文档与组织方式，而不是让角色约束退化为目录习惯。
- 新增文档入口时，先确认其是否只是导航 / 展示层；若它开始承载协议真相，应回退设计并保持 contract 中心不变。
