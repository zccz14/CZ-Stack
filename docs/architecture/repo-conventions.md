# 仓库组织约定

CZ-Stack 采用“**模块角色优先**”的仓库组织方式：我们关心的是模块承担什么职责、遵守什么依赖方向，而不是强制使用固定目录名。

## 角色优先，不要求固定目录名

- 不要求必须使用 `apps/`、`packages/`、`services/` 等传统分层目录名。
- 只要 workspace 发现规则、依赖方向与角色边界成立，模块可以放在任意等价物理目录中。
- 文档、脚本与配置应描述“角色约束”，而不是把目录名错误提升为架构本身。

这意味着：

- `web-app` 不因为位于 `modules/`、`apps/` 或其他目录而改变角色。
- `contract-package` 的关键不是物理路径，而是它是否仍然作为协议单一事实源。
- `api-service`、`cli-tool`、`docs-site`、`db-adapter` 都按角色职责判断，而不是按目录前缀判断。

## 首版为何选择 `modules/`

首版示例选择 `modules/` 作为主要物理容器，原因如下：

1. `modules/` 比 `apps/` / `packages/` / `services/` 更中性，能同时容纳应用、服务、契约、数据库边界与工具模块。
2. 它能提醒维护者：这里承载的是“不同角色模块”，而不是某一种固定分层范式。
3. 在模板早期阶段，使用一个统一容器可以减少 workspace 配置分散与目录命名争议。

但这只是**首版示例容器**，不是永久强约束。若未来迁移到其他目录结构，只要同步更新 workspace globs、文档链接与脚本入口，并继续满足角色边界，就不构成架构偏离。

## 与 workspace 规则的关系

当前仓库使用 `pnpm-workspace.yaml` 声明首版发现规则，例如：

- `modules/*`
- `tooling/*`
- `docs-site`

这些规则的作用是让首版模板可直接安装、检查与扩展，而不是宣告目录名不可变。后续若目录演进，应优先确保：

1. workspace 仍能发现对应模块；
2. 文档入口与命令示例保持一致；
3. 模块角色与禁止反向依赖规则不被破坏。

同时，workspace 发现规则不代表这些路径下已经存在已实现模块；它们只是为后续任务预留的发现入口。

## 维护约定

- 新增模块时，先声明其角色，再决定其物理位置。
- 调整目录时，先更新 workspace 配置与文档，再移动实现。
- 若某目录命名会误导角色理解，应优先修正文档与组织方式，而不是让角色约束退化为目录习惯。
