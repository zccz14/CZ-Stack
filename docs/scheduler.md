# 调度器

## 背景

1. 访谈 + 写 Spec + 重型开发流程，通常需要持续 1-2 小时。太慢。并行度不够高。
2. 执行 Task Session 已经实现闭环。只需要无脑推进即可迭代基线。

## 定义

**Task 是一个基线迭代增量。Task 的职责就是更新基线。**

调度器只有一个目标：尽快将基线推进到目标状态。

- 最大化并行度。通过拆分 Task，允许更多的并行度。
- 最小化人工介入。允许实际执行结果与无法言说的预期有偏差。

每个 Task 的固定执行单元是：

```
从最新本地基线分支开始 -> 新 Git WorkTree -> 基线验证 -> 探索恢复上下文 -> 实现 -> 测试 -> 提交 PR -> 跟进 checks/review 修复 -> merge -> 清理 WorkTree -> 刷新本地基线分支
```

每个 Task 应该从基线 + 一个 Spec 开始，无人值守地，最终将基线向前迭代推进一步。

**Task 自创建之后到结束，完全不含任何人工**。

Task 的成功的唯一标准是：PR 已 merge、对应 WorkTree 已清理、主工作区本地基线已刷新。三者缺一不可。

Task 的成功不意味着实现 100% 符合预期。

禁止 Task 层面进行任何人工决策；
禁止 Task 被阻塞在人工决策节点上；

任何传统意义上需要人工确认的新问题，都意味着这个 Task 的彻底失败。Task 的失败将调用特殊的 Tool，将失败信息整理反馈给上层 AI，由它来决定如何调整 Task，重新派发，或调整整体计划。Task 内部可以优化失败信息流程，或者更加审慎地汇报失败。

Task 的典型失败包括：“Spec 不清晰导致出现重大歧义，需要澄清”、“Spec 已经与当前基线方向冲突，无法靠后续基线推进自然恢复”等。但是不包括“PR check 未通过”，因为这可能是实现细节问题，或者基线状态变化引入的间接问题，这些都应该由 Task 内部的修复流程来解决，而不是直接上报失败。

## Task 的核心字段

- `task_id` (UUID)
- `task_spec` (TEXT)，即这个 Task 的 Spec 内容，应该是一个明确的、可执行的增量目标描述，基于最新的基线状态。内容格式见[定义](./task-spec.md)。
- `session_id` (NULLABLE TEXT)，即这个 Task 绑定的 Agent Session ID。用于派发执行。
- `worktree_path` (NULLABLE TEXT)，即这个 Task 使用的 Git WorkTree 绝对路径。用于管理执行环境。
- `pull_request_url` (NULLABLE TEXT)，即这个 Task 最终提交的 PR URL。用于追踪结果。
- `dependencies` (TEXT, JSON ARRAY)，即这个 Task 当前认定的候选前置 Task ID 列表。规划器可以给初始值，但执行层或基线验证允许完全覆盖它。调度器只将其作为软调度提示，不将其视为硬门禁。
- `done` (BOOLEAN)，即这个 Task 是否已经完成。注意，完成不等于成功，完成的定义是 Task 已经不再继续执行了，无论是成功还是失败。所有未完成的 Task 都仍然处于调度池中，但这不意味着调度器必须立即为它们全部创建 Session。调度器应优先继续已有的活跃 Task Session，并在基线更新时重新审视所有未完成 Task 是否值得再次验证或派发。
- `status` (TEXT)，即这个 Task 的当前状态的缓存结果，不需要通过 AI 恢复。正常来说，Task 的状态是可以通过 Agent 来推断的，但是这涉及异步动作，不可快速计算。因此这个字段是为了快速查询 Task 状态而设计的缓存字段。

不需要的字段：

- `spec_commit`: 产生 Spec 的分支的 commit hash。后文会解释为什么不需要这个字段。
- `base_commit`: 旧基线 commit 本身不足以判断 Spec 是否仍适用。
- `next_run_after`: 时间流逝本身不是有效的调度信号。
- PR checks / review / mergeability 等平台状态镜像：这些状态应按需查询，而不是常驻存库。

Task 的执行成功的标准：

1. `pull_request_url` 被正确填充，且对应的 PR 被 merge。(`gh pr view <pull_request_url> --json state` 返回 `{"state": "MERGED"}`)
2. `worktree_path` 被正确填充，且对应的目录不存在。
3. 主工作区已经完成本地基线刷新，不能停在“PR 已 merge 但本地基线还没刷新”的半完成状态。

### 关于 Spec 过期失败的判定

Spec 中包含了 Spec 的背景和前提假设。例如，由于现状是 A，我们遇到了 B 问题，因此我们需要做 C 来解决这个问题。这个 Spec 的前提假设是现状是 A，如果现状已经变成了 A'，那么这个 Spec 就过期了。

用一个过期的 Spec 来执行，大概率是失败的。我们可以尽早地令其失败：

在 Task 执行的早期阶段，会有一个基线验证的步骤，一方面需要验证基线是否能通过回归测试，另一方面则是需要验证 Spec 的前提假设是否仍然成立。

这里需要明确区分两种结果：

1. `Assumptions` 暂时不成立，但仍有可能随着后续基线推进而成立。这不是失败，而应进入 `waiting_assumptions`，并允许执行层重写 `dependencies`。
2. Spec 本身存在重大歧义，或者已经与当前基线方向冲突，无法靠后续基线推进自然恢复。这才是失败，应上报给上层规划器处理。

Task 中不包含产生 Spec 的 commit 信息。依赖 commit 来判断 Spec 是否过期的做法是不可行的。因为基线本身会并行地被多个 Task 推进，如果依赖 commit 来判断 Spec 是否过期，那么就只会有一个 Task 能够成功，其他 Task 都会因为 Spec 过期而失败，这样就完全没有并行度了。commit 信息自身实际上并没有有效的信息量，不如直接基于 Spec 的前提假设来判断是否过期。

另一个问题是：在 Task 执行的晚期，如果由于基线已经被其他 Task 推进，导致 Spec 事实上已经过期，这是有可能的。但这通常是不合理的，这意味着上层的规划器安排了两个有冲突的 Task 在同一时间并行执行，这本身就是一个规划错误。我们可以允许其中一个 Task 成功，其他 Task 失败，但是更好的做法是，在 Task 执行的早期阶段就通过验证 Spec 的前提假设来避免这种情况的发生。如果真的遇到了这种问题，合并前的验证失败就应当触发失败上报，而不是强行继续出站。

## 兜底调度逻辑

时间流逝不是调度信号。**基线变化才是调度信号。**

调度器的真实兜底逻辑，应当建立在“每次基线前进，所有未完成 Task 都重新获得一次验证机会”这一原则上，而不是建立在“过一段时间再试一次”上。

### 局部推进

如果某个 Task 已经拥有自己的 Task Session，并且该 Session 只是暂时 idle，那么调度器应优先继续这个 Task 的局部闭环推进。这属于对已有执行上下文的复用，不应轻易打断。

### 全局兜底

每当一个 PR 被 merge，基线向前推进一次，调度器都应把这次基线变化视为一次全局重检机会：

1. 对所有 `done = false` 的 Task，重新考虑是否值得触发一次基线验证。
2. 这里的“重新考虑”不是平均用力地同时启动所有 Task Session，而是先排序，再优先验证最可能变 ready 的 Task。
3. 重新验证的目标不是判断“依赖有没有全部完成”，而是判断“当前最新基线能否通过该 Task Spec 的 `Assumptions` 验证”。

### 排序启发式

在基线更新后，调度器可以按以下启发式优先排序：

1. `dependencies` 中包含刚刚成功推进基线的 Task ID。
2. 当前 `dependencies` 数量为 `0` 的 Task。
3. 当前 `dependencies` 数量较少的 Task。
4. 其他尚未完成的 Task。

这里的 `dependencies` 仍然只是软提示，不是硬门禁。它的作用是帮助调度器决定：在基线刚刚变化之后，哪些 Task 更值得先花 token 去验证。

### 复杂度判断

如果系统中有 `N` 个 Task，并且最终也会有数量级接近 `N` 的基线推进，那么在最坏情况下，每次基线更新都触发一次对所有未完成 Task 的重新验证，整体复杂度大致是 `O(Task^2)`。

这个复杂度在当前目标下是可以接受的：

- 每个 commit 都来自于某个 Task，本身已经代表了一次有效基线推进。
- 验证成本相比错误地串行化整个系统，代价更低。
- 平均情况通常会更好，因为会有不少 Task 在验证后直接进入执行，而不是长期留在等待池里。

### 规划器的再介入

调度器负责短周期、确定性的兜底推进；规划器负责结构性纠偏。

因此，在基线更新后，除了调度器做兜底扫描之外，我们总是可以再启动一个规划器，重新研究现存 Task 是否需要调整，尤其关注：

- 长时间停留在 `waiting_assumptions` 的 Task。
- 被基线验证反复打回的 Task。
- 已经明显不适合当前基线结构的 Task。

调度器负责“再试一次”，规划器负责“是不是任务本身该改了”。

## Task 的拆分与 Task 之间的关系

任务依赖表示**当前 Task 认为，哪些前置任务的基线增量最可能帮助自己的 Assumptions 成立，或显著降低返工概率**；依赖不是任务相关性的表达，也不是硬门禁，而是基线前置条件的软提示表达。规划器可以提供一份初始依赖 DAG 快照，但执行层和基线验证流程允许在运行时完全重写它。

典型、直观地理解 Task 拆分的例子：一个涉及前端、后端、接口变更的功能开发，应该拆成多个 Task：先做 contract，再并行做 producer 和 consumer，最后做 integration。

但是这种拆分方法太粗糙。我们需要更细化的拆分原则和明确的依赖关系定义，来指导上层 AI 如何拆分 Task，组织成 DAG。

定义上，每个 Task 都是基线的增量，具有完备的输入（基线 + Spec）和输出（PR MERGED 产生新的基线）。

Task 的尺寸应当适当，既不能太大导致并行度不足，也不能太小导致管理开销过大。一个合理的尺寸是，Agent 完成一个 Task 的时间通常在 10-30 分钟之间。

每个 Task 要能合并 PR，按照 TDD 流程来说，必须产生新的测试用例。过细的 Task 可能会导致测试用例的粒度也过细，用例会变成实现的简单重复，失去测试的意义。过粗的 Task 则可能导致中间应该被测试的用例被遗漏，导致实现质量不够。

Task 的典型例子：

- 实现一个具体的函数、组件。测试用例也针对这个函数、组件的输入输出进行设计。

## Spec vs Implementation Plan

Implementation Plan 是在 Spec 基础上的更细化的执行方案，包含了更具体的步骤和细节。

如果说 Spec 是给 Senior Developer 看的，侧重于“做什么”和“为什么”，那么 Implementation Plan 则是给 Junior Developer 看的，侧重于“怎么做”。

Implementation Plan 的必要性是，降低实现的不确定性。但是这个 Plan 没有必要进行人工 Review，它太过于细节。它属于某个 WorkTree 内部，主要目的包括快速恢复中断的任务执行，以及允许交给能力更弱的 LLM 去执行。
