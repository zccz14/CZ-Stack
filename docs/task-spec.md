# Task Spec 的定义

格式上是个 Markdown 文档，但保存在 SQLite 的 spec 字段中。内容上应该包含：

```markdown
# Title

## Assumptions

## Goal vs Non-Goal

## Core Path

## Value Alignment
```

注意：Spec 不包含“什么算成功”、“什么情况必须失败上报”、“失败后如何反馈”。这是全局统一的协议，不是每个 Task 都需要单独定义的内容。

## 1. 标题 (Title)

一句话。要写成“基线增量目标”，不是泛泛主题。能自然映射成一个 PR 标题。

## 2. 前提假设 (Assumptions)

可验证的基线现状列表。不要写背景故事，只写可验证事实。最好每条都尽量能被只读检查确认。这是后续失效检测的基础。

优先写“外部可观测事实”。例如“某个概念尚不存在”、“某个流程当前停在某阶段”。少写“代码大概是这样组织的”这类容易漂移的描述。

## 3. 目标与非目标 (Goal vs Non-Goal)

**正反两面，严格框住 scope，不允许遗漏 / 蔓延。**

非目标写得和目标一样具体，甚至比目标更多更具体。

非目标不代表完全不能做，而是尽量不碰、少碰、简化处理。如果在执行过程中遇到了需要做非目标里内容的情况，应该优先考虑不做，或者做一个非常简陋的版本，来满足“尽快推进基线”的核心目标。只有当完全无法推进基线了，才允许做最小桥接改动来打通实现。

这里的“最小桥接改动”是指：只为恢复主路径推进能力而做的最小必要改动，不得顺势扩展 scope，不得引入新的外部概念、接口面或额外承诺。执行层应优先选择一个粗糙但可合并的打通方案，而不是借机把非目标做完整。

## 4. 核心路线 (Core Path)

**核心路线，能作为下层的导向。下层及时回归核心路线，避免过度偏离，徒增寻路开销。**

核心路线本质上是推理链路的展示。是设计方案。是实现目标的路径。

建议包含:

- 当前概念格局
- 目标概念变化与主路径
- 备选路径及不选原因

因为我们已经有了现状和目标，核心路线要展示从现状到目标的推理链路。每一步都要清晰、合理、充分。整体上要尽量简洁，不要冗余。每一步都要扣在明确的核心概念与关系上，不能停留在文件或者代码的层面。要让执行层能够理解为什么要这么做，以及每一步的价值是什么。

例如：我们现在有 A, B, C 这些概念，定义分别是什么。我们的目标是新增 D 这个概念，或者修改 A 的定义。为什么我们需要新增 D？为什么我们需要修改 A？新增 D 或修改 A 能带来什么价值？新的 D 和 A、B、C 之间是什么关系？

在 Spec Review 阶段，评审者需要重点关注这个推理链路是否清晰、合理、充分、简洁。并且还需要评估这个 Spec 的核心路线的工作量是否足够小，是否在预定的 10～30 分钟的实现范围内。

禁止核心路线中关注文件细节的变化。

涉及的业务概念与原因：盯住核心概念与关系的变化，而不是文件本身的变化，这里还需要比较强的推理过程。这里的“概念”既可以是业务概念，也可以是调度、状态、契约、依赖、执行单元等基础设施概念。执行层要理解：为什么这个增量值得单独成为一个 Task。

以及，备选路径的分析：提出并指明为什么不选择其他路径。要展示我们对其他路径的分析和比较，说明为什么我们选择了这个路径，而不是其他路径。

## 5. 价值观对齐 (Value Alignment)

**价值观对齐，能带给下层的自由裁量权，是无人值守的最后一道防线。**

如果实际情况有冲突或者分歧，允许执行层自由裁量，但是需要明确在遇到什么问题的时候，获得价值 A 比获得价值 B 更重要。有个优先级排序。如果执行层无法从价值观对齐中获得充足的证据解决分歧。仍然会执行失败。

执行层自由裁量的落点是 Implementation Plan，不影响 Spec 的稳定性。

要形成足够稳定的价值观对齐，在制定 Spec 的时候，访谈就应该尽量挖掘出潜在的分歧点（风险），并且让被访谈者明确表达在这些分歧点上，什么样的结果是更好的（缓解）。

价值观对齐不应只写抽象口号，而应尽量写成“冲突场景 + 价值排序”的形式，让执行层在遇到真实分歧时有证据可依。例如：

- 当“保持基线可持续推进”和“局部实现完美”冲突时，优先前者。
- 当“避免人工决策节点”和“尝试覆盖更多边界情况”冲突时，优先前者。
- 当存在多目标排序时，应明确写出稳定顺序，例如：保持部署简单 > 保持实现简单 > 尝试覆盖更多边界情况。

如果价值观对齐只能表达一般性的偏好，而不能覆盖可预见的冲突场景，那么这个 Spec 仍然是不充分的。执行层无法从中获得充足证据解决分歧时，仍然会执行失败。
